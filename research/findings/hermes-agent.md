## 2. TL;DR

- Hermes is a **production-grade personal agent** (not a research artifact) whose headline differentiator is a **closed self-improvement loop over a Skills library**: skills are markdown files (`SKILL.md` + `references/`/`templates/`/`scripts/`) — the same "Agent Skills" format as Claude Code / agentskills.io — and the agent **creates, patches, and consolidates them autonomously from its own conversation history**.
- The core loop is *not* the evolutionary "propose→test-in-isolation→keep-if-better" pattern our seed-AI uses. It is a **reflection/distillation loop**: after a turn, a **forked agent** (daemon thread, restricted to memory+skill tools) re-reads the transcript and asks "did anything happen worth saving as procedural knowledge?" There is **no verifier / no benchmark / no A-B retention gate** — retention is decided by an LLM judgment, not by measured improvement.
- The most reusable, directly transferable assets are: (1) the **verbatim self-improvement review prompts** (memory vs. skill vs. combined), which are unusually well-thought-out about *what NOT to learn* (anti-patterns that "harden into refusals"); (2) the **curator** — a separate background process that consolidates/prunes/de-duplicates the skill library at scale with usage telemetry; (3) the operational engineering for **running the reflection cheaply** (forked agent inherits the parent's cached system prompt to hit the prefix cache → ~26% cost cut) and **safely** (tool whitelist, provenance ContextVar so it never edits user-authored or bundled skills).
- There is also an `optional-skills/research/darwinian-evolver/` skill — the closest thing to true evolutionary search in the repo — but (as of inspection) it appears to be a *skill/playbook* (instructions + a helper script), not a built-in engine. Worth flagging, examined below.
- Honest signal: **medium-high for our purposes.** Hermes is the best open-source reference for the *memory/skill-as-procedural-knowledge* and *agent-curates-its-own-knowledge-base* dimensions, and for the production plumbing of long-horizon, multi-surface autonomous agents. It is **weak** on the verification/evolutionary-selection dimension that is the spine of a seed AI — it teaches what to borrow for the "memory + self-distillation" half, and is a cautionary example for the "keep-only-if-verifiably-better" half (Hermes deliberately does *not* gate on verification, and its own prompts admit the failure mode of learned constraints going stale).

## 3. What it does & how it works

### 3.1 Product surface (so we can judge relevance)
Hermes is a single agent process exposed through (a) a full terminal TUI (`hermes`), and (b) a **gateway** that bridges Telegram/Discord/Slack/WhatsApp/Signal/Email (`gateway/`). It is model-agnostic (`/model provider:model`), runs on six "terminal backends" (local, Docker, SSH, Singularity, Modal, Daytona — the last two give serverless hibernation), has a **cron scheduler** (`cron/scheduler.py`) for unattended automations, can **spawn isolated subagents** for parallel workstreams (`tools/delegate_tool.py`), and ships **batch trajectory generation + a trajectory compressor** (`batch_runner.py`, `trajectory_compressor.py`, 65KB) explicitly "for training the next generation of tool-calling models." That last point matters: Nous is using the agent as a **data-generation flywheel** for model training, which is a different (and complementary) notion of "self-improvement" than per-skill distillation.

### 3.2 The four advertised "learning loop" components
The README states it: "Agent-curated memory with periodic nudges. Autonomous skill creation after complex tasks. Skills self-improve during use. FTS5 session search with LLM summarization for cross-session recall. Honcho dialectic user modeling." Mapped to code:

1. **Skills as procedural memory** — `tools/skill_manager_tool.py` exposes a `skill_manage` tool with actions `create / edit / patch / delete / write_file / remove_file`. Skills live in `~/.hermes/skills/<category>/<name>/SKILL.md`, with optional `references/ templates/ scripts/ assets/`. The tool's own description encodes the *policy* of when to learn: "Create when: complex task succeeded (5+ calls), errors overcome, user-corrected approach worked, non-trivial workflow discovered… Update when: instructions stale/wrong, OS-specific failures, missing steps or pitfalls found during use."
2. **The background self-improvement review** — `agent/background_review.py` + `AIAgent._spawn_background_review` (`run_agent.py:1378`). After a turn, a forked agent replays the transcript and decides what to persist. (Mechanism detail in §3.3.)
3. **Agent-curated memory with nudges** — declarative memory in `MEMORY.md` / `USER.md` via a `memory` tool (`agent/memory_manager.py`), plus periodic "nudges" driven by `_memory_nudge_interval`. External memory providers (Honcho "dialectic" user modeling, mem0, supermemory) plug in via `agent/memory_provider.py`.
4. **The curator** — `agent/curator.py` (78KB) + `hermes_cli/curator.py` + `tools/skill_usage.py`: a *separate* background maintenance process that consolidates overlapping skills, prunes stale/unused ones, and keeps the library at "class level" rather than a flat list of one-session entries. Uses usage telemetry and a pin/archive model. (Detail in §3.4.)

### 3.3 The self-improvement review loop — exact mechanism
Trigger (in `agent/conversation_loop.py`, the real call site — `run_agent.py` only defines the spawner):
- **Memory review** fires when `agent._turns_since_memory >= agent._memory_nudge_interval` (counted per *user turn*; reset to 0 on fire) — `conversation_loop.py:555-562`.
- **Skill review** fires when `agent._iters_since_skill >= agent._skill_nudge_interval` AND `"skill_manage" in agent.valid_tool_names` — `conversation_loop.py:4779-4785`. Note it counts **tool iterations this turn**, so heavier/more agentic turns are the ones that trigger skill distillation. This is a deliberately cheap heuristic proxy for "this turn was complex enough to be worth learning from."
- Both only fire `if final_response and not interrupted`, and run **after** the user's response is delivered so they "never compete with the user's task for model attention" (`conversation_loop.py:4795-4805`).

Execution (`agent/background_review.py::_run_review_in_thread`):
- Spawns a **daemon-thread forked `AIAgent`** with `max_iterations=16`, `quiet_mode=True`, `skip_memory=True`, inheriting the parent's `provider/model/base_url/api_key/credential_pool` (so OAuth/session creds work) and — critically — `review_agent._cached_system_prompt = agent._cached_system_prompt`. The comment: inheriting the cached prompt makes the fork's HTTP request "hit the same Anthropic/OpenRouter prefix cache the parent warmed… ~26% end-to-end cost reduction on Sonnet 4.5" (lines 433-442, citing issue #25322 / PR #17276).
- **Tool whitelist**: `set_thread_tool_whitelist(...)` restricts the fork to the `memory` and `skills` toolsets; everything else is denied at runtime with "Background review denied non-whitelisted tool" (lines 459-472).
- **Safety**: installs a `_bg_review_auto_deny` approval callback so any dangerous-command guard resolves to "deny" rather than blocking on `input()` and deadlocking the TUI (lines 347-356).
- **Provenance**: `review_agent._memory_write_origin = "background_review"`. `tools/skill_provenance.py` exposes a `ContextVar` so that `skill_manage(create)` only calls `mark_agent_created(name)` when `is_background_review()` is true — i.e. **only autonomously-created skills are owned by the curator; user-directed foreground skill writes are never auto-curated** (`skill_manager_tool.py:880-891`).
- Output: the fork's successful tool actions are summarized and surfaced to the user as `💾 Self-improvement review: <summary>` (lines 505-528). Main conversation and prompt cache are untouched.

The **prompts themselves are the load-bearing IP** and are quoted verbatim in §4.

### 3.4 The curator (consolidation/pruning at scale)
Whereas the per-turn review *adds* knowledge, the curator *keeps the library healthy*. Detail and excerpts in §4.2, but in summary it: classifies agent-created skills (consolidate vs. prune), respects `pinned` (block deletion only) and `agent_created` flags, uses `tools/skill_usage.py` telemetry (`patch_count`, usage, `mark_agent_created`, `forget`), and uses the `absorbed_into` field on delete to distinguish "merged into umbrella X" from "truly pruned" so downstream references (e.g. cron jobs naming the old skill) can be rewritten.

## 4. Evidence from the code

### 4.1 The skill-management tool (agent's write path to procedural memory)
`repo@SHA:tools/skill_manager_tool.py`. Actions and guards:
- `create / edit / patch / delete / write_file / remove_file`; skills validated for YAML frontmatter (`name`, `description` required), name regex `^[a-z0-9][a-z0-9._-]*$`, size caps (`MAX_SKILL_CONTENT_CHARS = 100_000`, `MAX_SKILL_FILE_BYTES = 1 MiB`). All writes are **atomic** (`tempfile` + `os.replace`) with **rollback** on security-scan block.
- `patch` reuses a **fuzzy find-and-replace engine** (`tools/fuzzy_match.fuzzy_find_and_replace`) so the model doesn't fail on whitespace/indentation mismatches — and on failure returns a `file_preview` + `format_no_match_hint` so the model can self-correct (lines 604-628). This is a nice ergonomic detail for any agent that edits its own files.
- The tool **schema description is itself the learning policy** (verbatim, lines 900-931):
  > "Skills are your procedural memory — reusable approaches for recurring task types… Create when: complex task succeeded (5+ calls), errors overcome, user-corrected approach worked, non-trivial workflow discovered, or user asks you to remember a procedure. Update when: instructions stale/wrong, OS-specific failures, missing steps or pitfalls found during use. If you used a skill and hit issues not covered by it, patch it immediately… Good skills: trigger conditions, numbered steps with exact commands, pitfalls section, verification steps."
- Pin semantics (`_pinned_guard`): pin blocks **deletion only**; patch/edit still allowed. The agent is told to redirect the user to `hermes curator unpin <name>`.
- `absorbed_into` on delete: must name an existing umbrella skill (validated) or be `""` for a true prune — "lets the curator tell consolidation from pruning without guessing, so downstream consumers (cron jobs that reference the old skill name, etc.) get updated correctly."

### 4.1b The self-improvement REVIEW prompts (verbatim — high value)
`repo@SHA:agent/background_review.py`. Three prompts; the skill one (`_SKILL_REVIEW_PROMPT`, lines 45-148) is the most instructive. Selected verbatim passages:

> "Review the conversation above and update the skill library. Be ACTIVE — most sessions produce at least one skill update, even if small. A pass that does nothing is a missed learning opportunity, not a neutral outcome."

> "Target shape of the library: CLASS-LEVEL skills, each with a rich SKILL.md and a `references/` directory for session-specific detail. Not a long flat list of narrow one-session-one-skill entries."

Preference order (which existing artifact to extend before creating new): "1. UPDATE A CURRENTLY-LOADED SKILL… 2. UPDATE AN EXISTING UMBRELLA… 3. ADD A SUPPORT FILE under an existing umbrella… 4. CREATE A NEW CLASS-LEVEL UMBRELLA SKILL when no existing skill covers the class." The new-skill name "MUST NOT be a specific PR number, error string, feature codename, library-alone name, or 'fix-X / debug-Y / audit-Z-today' session artifact."

The **anti-learning section is the cleverest part** — it explicitly enumerates what must NOT be learned because it "become[s] persistent self-imposed constraints that bite you later when the environment changes":
> "• Environment-dependent failures: missing binaries, fresh-install errors, post-migration path mismatches, 'command not found', unconfigured credentials, uninstalled packages…
> • Negative claims about tools or features ('browser tools do not work', 'X tool is broken', 'cannot use Y from execute_code'). These harden into refusals the agent cites against itself for months after the actual problem was fixed.
> • Session-specific transient errors that resolved before the conversation ended. If retrying worked, the lesson is the retry pattern, not the original failure.
> • One-off task narratives…"
> "If a tool failed because of setup state, capture the FIX (install command, config step, env var to set)… never 'this tool does not work' as a standalone constraint."

The memory prompt (`_MEMORY_REVIEW_PROMPT`) is narrowly about the *user* ("Has the user revealed things about themselves… expectations about how you should behave"). The split is explicit: **memory = "who the user is and what the current situation/state of your operations are"; skills = "how to do this class of task for this user."**

### 4.2 The curator (`repo@SHA:agent/curator.py`, 78KB)
Header invariants (lines 1-20): "inactivity-triggered (no cron daemon)… Only touches agent-created skills… **Never auto-deletes — only archives. Archive is recoverable.** Pinned skills bypass all auto-transitions. Uses the auxiliary client; never touches the main session's prompt cache."

Lifecycle + cadence constants (lines 56-59): `DEFAULT_INTERVAL_HOURS = 24*7` (7 days), `DEFAULT_MIN_IDLE_HOURS = 2`, `DEFAULT_STALE_AFTER_DAYS = 30`, `DEFAULT_ARCHIVE_AFTER_DAYS = 90`. `apply_automatic_transitions()` (line 268) walks managed skills and moves them active→stale→archived by last-activity timestamp; pinned skills are skipped; archive is a `mv` into `~/.hermes/skills/.archive/`.

`CURATOR_REVIEW_PROMPT` (lines 357-491) drives a **forked auxiliary agent** to do *umbrella-building consolidation*. Notable design decisions, verbatim:
> "A collection of hundreds of narrow skills where each one captures one session's specific bug is a FAILURE of the library — not a feature. An agent searching skills matches on descriptions, not on exact names; one broad umbrella skill with labeled subsections beats five narrow siblings for discoverability."

> "4. DO NOT use usage counters as a reason to skip consolidation. The counters are new and often mostly zero. Judge overlap on CONTENT, not on use_count. 'use=0' is not evidence a skill is valuable; it's absence of evidence either way."

> "5. DO NOT reject consolidation on the grounds that 'each skill has a distinct trigger'. Pairwise distinctness is the wrong bar. The right bar is: 'would a human maintainer write this as N separate skills, or as one skill with N labeled subsections?'"

Three consolidation moves: (a) merge into existing umbrella, (b) create new umbrella, (c) demote to `references/`/`templates/`/`scripts/`. It enforces **package integrity** (don't flatten a SKILL.md whose body links to support files and orphan those files). It demands a machine-readable result block:
```yaml
consolidations:
  - from: <old-skill-name>
    into: <umbrella-skill-name>
    reason: <one short sentence — why merged, not just 'similar'>
prunings:
  - name: <skill-name>
    reason: <one short sentence — why archived with no merge target>
```
There is also a **`CURATOR_DRY_RUN_BANNER`** (lines 330-354): a report-only preview mode that produces the *same* YAML but takes no mutating action, gated behind human approval (`hermes curator run` to go live). This is a clean pattern for "let the agent propose destructive maintenance, but require a human/second check before it acts."

### 4.2b Skill-usage telemetry (`repo@SHA:tools/skill_usage.py`)
The data structure backing curation. Record schema (`_empty_record`, line 430): `{use_count, view_count, patch_count, state ∈ {active,stale,archived}, pinned, created_by/agent_created, timestamps}`. Helpers: `bump_view/bump_use/bump_patch`, `mark_agent_created`, `forget`, `is_agent_created/is_bundled/is_hub_installed`, `is_curation_eligible`, `archive_skill`, a suppression list so archived built-ins stay archived across `hermes update` re-seeds. `activity_count` sums use+view+patch. Stored as a lock-guarded JSON file under the skills dir.

### 4.3 Skill retrieval / progressive disclosure (`repo@SHA:agent/prompt_builder.py:build_skills_system_prompt`)
Skills are surfaced to the model via **progressive disclosure**, the same model as Claude Code / agentskills.io: the system prompt contains only a **compact index** — skill *name + one-line description, grouped by category* (lines 1104-1153) — NOT the full SKILL.md bodies. The agent then pulls a full body on demand with `skill_view` / `/skill-name`. The index is double-cached: an in-process LRU keyed by (skills_dir, tools, toolsets, platform, disabled-set) plus a disk snapshot `.skills_prompt_snapshot.json` validated by an mtime/size manifest (lines 1059-1102) so it survives restarts and keeps the system-prompt prefix byte-stable for prompt-cache hits. Skills can declare `conditions`/`platforms` so the index only shows skills compatible with the current toolset/OS (`_skill_should_show`, `skill_matches_platform`). `SKILLS_GUIDANCE` (in `hermes_constants`) is the static block telling the model how to use the index.

### 4.4 The `darwinian-evolver` skill — the only *evolutionary* artifact (`repo@SHA:optional-skills/research/darwinian-evolver/`)
This is **not** part of Hermes's core loop. It is an *optional skill* (playbook + helper scripts) that installs and drives **Imbue's external `darwinian_evolver`** (AGPL-3.0) tool — "an LLM-driven evolutionary search loop — to optimize a prompt, regex, SQL query, or small code snippet against a fitness function." It is explicitly invoked only via subprocess/CLI ("mere aggregation"; "Do NOT import upstream classes into Hermes itself"). It is high-signal for *our* project because the SKILL.md cleanly documents the organism/evaluator/mutator schema:
- **`Organism`** — Pydantic model holding the artifact being evolved (`prompt_template`, `regex_pattern`, `code_block`, …) + a `run()` method.
- **`Evaluator.evaluate(organism) -> EvaluationResult(score∈[0,1], trainable_failure_cases, holdout_failure_cases, is_viable)`** — note the explicit **held-out failure cases kept out of the mutator's view to detect overfitting**, and `is_viable` gate (a 0-score viable organism is fine; non-viable means "nothing to evolve from").
- **`Mutator.mutate(organism, failure_cases, learning_log_entries) -> list[Organism]`** — LLM proposes fixes from failures; returns `[]` on parse failure (loop tolerates).
- Hyperparameters table (`--num_iterations`, `--num_parents_per_iteration`, `--verify_mutations` for cost saving, `--midpoint_score=p75`, `--sharpness`). References Darwin Gödel Machines (arXiv 2505.22954) and PromptBreeder (arXiv 2309.16797) — the actual evolutionary-agent literature. (The orchestrator may want dedicated sub-agents on Imbue's darwinian_evolver and DGM; flagging, not pursuing here.)

## 5. What's genuinely smart

1. **Skills as the unit of self-improvement, in plain markdown.** By making the learnable artifact a `SKILL.md` file (frontmatter + body + `references/`/`templates/`/`scripts/`), Hermes gets: human-auditable diffs, trivial versioning (git/patch), composability, and the ability for the *same* agent to both author and consume them. The artifact is procedural ("how to do this class of task"), distinct from declarative memory ("who the user is"). This separation is principled and reusable.
2. **A reflection loop that is cheap and safe by construction.** The forked-agent design (daemon thread, runs after the user's response, restricted toolset, inherits the parent's cached system prompt for prefix-cache reuse, auto-denies dangerous commands, provenance ContextVar) is a textbook example of how to bolt an autonomous background self-modification process onto a live agent without (a) blowing cost, (b) corrupting the main conversation/cache, or (c) letting it run unsafe tools. The ~26% measured cost reduction from cache-prefix inheritance is a concrete, transferable engineering win.
3. **The anti-learning rules.** The single most valuable idea for a *self-improving* system: explicitly teaching the distiller **what not to persist** because environment-dependent failures and "tool X is broken" claims "harden into refusals the agent cites against itself for months." Any system that writes to its own long-term memory/skills needs exactly this guardrail, or it accumulates stale self-imposed constraints. This is a failure mode our seed AI will hit, and Hermes has a battle-tested prompt for it.
4. **Two-tier maintenance: add (per-turn review) vs. curate (periodic curator).** Splitting "capture new knowledge eagerly" from "keep the library coherent at scale" is smart. The per-turn reviewer is told to be *active* (bias toward capturing); the curator counteracts the resulting sprawl by consolidating into class-level umbrellas. The curator's "judge overlap on CONTENT not use_count" and "pairwise distinctness is the wrong bar" instructions are subtle, correct, and hard-won.
5. **Recoverable-by-default destructive ops + dry-run gating.** Curator archives (never deletes), keeps archives recoverable, and offers a report-only dry-run that emits the same structured plan for human approval. Good pattern for any agent allowed to prune its own knowledge.
6. **Progressive disclosure + aggressive caching for the skill index.** Only names+descriptions in the prompt, bodies loaded on demand, with a manifest-validated disk snapshot — keeps a large skill library affordable and prefix-cache-friendly. Directly relevant to "how does a long-horizon agent carry a growing memory without paying for all of it every turn."
7. **The agent as a training-data flywheel.** `batch_runner.py` + `trajectory_compressor.py` exist to generate and compress trajectories "for training the next generation of tool-calling models." This is a *second*, orthogonal self-improvement axis (improve the base model, not just the skills) — relevant if our seed AI ever closes the loop from agent behavior back to model weights.
8. **Cache discipline as a first-class concern.** Recurring throughout: the review fork inherits the cached system prompt; the skill index is snapshot-cached and kept byte-stable; skill *slash commands* are injected as a **user message, not the system prompt, specifically to preserve prompt caching** (`AGENTS.md:175`). For a token-unlimited seed AI cost matters less, but the *technique* (keep the system-prompt prefix immutable so every call hits the provider KV cache) is worth knowing.

## 6. Claims vs. reality / limitations / critiques

**Claim framing.** Marketing says "the self-improving AI agent… the only agent with a built-in learning loop." Reality (well-stated by an independent walkthrough, Vikas Sah/Medium): "It does **not** update model weights. It is **not** fine-tuning itself in the background. Instead, Hermes has a practical agent-system learning loop" — improvement happens in *context/skills/memory*, not parameters. So "self-improving" = self-editing its prompt-time procedural knowledge. That is real and verified in code, but it is a *weaker* and *different* claim than "recursive self-improvement."

**The central structural critique — self-grading with no external verifier.** This is the most important point for our project, and it is corroborated by Hermes' *own issue tracker*. GitHub issue **#25833** ("Self-created skills lack mechanism-level guarantees for correctness and execution consistency") states it directly: *"The agent is simultaneously the author, executor, and quality inspector of its own skills. There is no external validation point or consistency check."* The DEV.to "Self-Trust Problem" analysis builds on this: a skill loop without external validation **"does not compound quality. It compounds confidence."** Concretely:
- **Skill rot / silent staleness.** A skill encodes an assumption ("the orders API returns total_price"); the API changes; the agent trusts its own skill and executes the stale procedure, producing confidently-wrong output. Skills carry **no `last_verified_at`, no `created_with_model_version`, no expiry metadata** (verified: the `skill_usage` record has timestamps for activity but no verification/version field; SKILL.md frontmatter requires only name+description).
- **Drift = overfitting.** "The agent refines a skill mid-use based on one noisy success… Self-improvement and overfitting are the same gradient pointed in hopefully-good directions." Nothing in the loop measures whether the *edit* actually improved anything — there is no A/B, no regression test, no held-out eval. (Contrast: the external `darwinian-evolver` skill *does* have holdout cases — but that's a separate optional tool, not the core loop.)
- **Conflict / island isolation.** GitHub issue **#12877** ("Serious systemic defects in the skills system: no pruning, island isolation, and missing conflict detection") reports overlapping skills that contradict each other, with which one fires being "a function of retrieval order, not intent." (The curator, added later in v0.12.0, is the partial answer to "no pruning" — but consolidation is itself an LLM judgment, not a correctness check.)

**Memory hygiene is advisory-only.** GitHub issue **#20595** ("Memory hygiene checks are advisory-only — bloat recurs under completion bias"): the memory tool's hygiene checks "return a `hygiene_warning` field but persist the entry anyway. In practice, models ignore the warning and continue." So the much-touted "agent-curated memory" still bloats because the guardrails are soft prompts, not hard gates. This is the same pattern as the skill loop: **policy lives in prompts, not in enforced mechanism.**

**The anti-learning prompt is a patch, not a fix.** The (now fixed) issue **#6051** documented the agent encoding transient failures as permanent "this tool doesn't work" skills. The fix was the prompt language quoted in §4.1b. #25833 correctly notes this is a prompt patch over an unchanged mechanism: "Prompt instructions can drift. Future changes can override them. The structural gap remains." Important lesson for us: **prompt-level guardrails on a self-modifying loop are fragile; they degrade as the model/prompts change.**

**Runtime fault-tolerance gap.** Issue **#22112** (per the DEV.to analysis): the 90-turn cap (`cli.py:419 max_turns=90`, shared with subagents) is a coarse "deterministic circuit breaker," but at the execution layer the agent "does not classify [repeated equivalent failures] and escalate. It retries the same sequence, silently consuming context and API budget until the turn cap terminates it." The takeaway for any long-horizon loop: you need **failure *classification* and escalation**, not just a global iteration cap. (There is *some* movement here — `agent/error_classifier.py` (53KB) and a `failure_limit` circuit breaker referenced at `conversation_loop.py:4485` exist — but the critique is that it's insufficient at the low level.)

**Compounding cost/tail-risk.** The Devdigest "Hidden Costs" analysis (engineering-serious, not a hit piece): skill-library bloat *erodes* the per-task savings (their illustrative figure: ~$0.024 → ~$0.039 per run as the library grows to ~200 skills because the agent spends tokens deciding which skill applies and sometimes fires the wrong one), and autonomy multiplies *run count* (a runaway recursive night of subagents spawning subagents → a surprise bill). Mitigations recommended: version-control the skills dir, skill expiry/auto-archive, daily spend caps + circuit breakers, weekly skill audits, least-privilege scoping, anomaly logging on skill writes. For a token-unlimited seed AI the *cost* angle is muted, but "skill bloat dilutes retrieval and fires the wrong skill" is a **real capability problem**, not just a billing one.

**Trust surface / persistence as attack vector.** Devdigest: "A poisoned input that convinces the agent to write a malicious skill doesn't vanish at end-of-session. It's now a file that loads every time the relevant trigger appears. Memory and skills turn a one-shot prompt injection into a durable foothold." Hermes mitigates with a security scanner (`tools/skills_guard.py`, scans hub installs always; agent-created only when `skills.guard_agent_created` is on — **off by default**, with the rationale that "the agent can already execute the same code paths via terminal() with no gate"). That rationale is honest but means **agent-authored skills are not scanned by default**.

**A specific secondary-source ERROR worth flagging.** The DEV.to "Self-Trust Problem" post (and its summary) repeatedly claim Hermes contains a **"GEPA self-evolution engine presented at ICLR 2026"** (Genetic-Pareto Prompt Evolution, citing Agrawal et al. 2025, "19% improvement with ~35× fewer rollouts"). **I could not find any GEPA implementation in the code.** A full-tree case-insensitive search for `gepa` / `genetic-pareto` / `pareto-prompt` across all `.py`/`.md`/`.yaml` at `@d1367355` returns **zero real matches** (only substring false-positives like "messa**gePa**yload"). The actual self-improvement mechanism is the **reflection/distillation loop** documented above (forked reviewer + curator), which is *not* GEPA and does *not* do Pareto-front prompt evolution against a scored eval. **Treat the "GEPA engine" claim as unverified/likely incorrect.** (GEPA itself is a real, separately-published technique and may be worth its own source review — but it is not, as far as the code shows, *in* Hermes.) Likewise the release notes call the upgraded skill-review prompt "rubric-based," but I found no literal scoring rubric in `background_review.py`/`curator.py` — it's a structured natural-language prompt with a preference order, not a numeric rubric.

**Reproducibility note.** There are no published benchmarks from Nous for the learning loop. The "20+ skills → 40% faster" and per-run cost figures appear only in secondary blog posts (DEV.to, Devdigest) with no methodology; treat as illustrative, not measured. What *is* verifiable is the mechanism and the prompts (in code).

## 7. Relevance to a self-improving, evolutionary agent

Hermes is **directly relevant to the memory/skill/long-horizon half** of our seed AI, and a **useful cautionary contrast on the verification/evolution half**.

- **Procedural-memory / skill library (HIGH relevance).** The "skills as plain-markdown, agent-authored, human-auditable, progressively-disclosed procedural memory" pattern is exactly a mechanism a seed AI could use to *accumulate and reuse what works* across its build loop — and to "improve its own prompts/tools" by editing skill files. The `skill_manage` action set (create/patch/edit/delete/write_file + `absorbed_into`), the validation/atomic-write/rollback plumbing, and the fuzzy-patch ergonomics are all reusable. Tied to: **self-modification of the agent's own playbook; tool/skill acquisition.**
- **Background reflection without blocking the main loop (HIGH).** The forked-agent-on-a-daemon-thread pattern (restricted toolset, post-response timing, cache-prefix inheritance, dangerous-command auto-deny, provenance tagging) is a clean blueprint for "the agent periodically steps back and updates its own memory/skills" — useful for the *meta* level of a seed AI that improves its own prompts/tools between build episodes. Tied to: **long-horizon running, self-improvement of prompts/tools, orchestration/safety.**
- **Two-tier add-vs-curate (MEDIUM-HIGH).** Eager capture (per-turn reviewer biased to "be active") + periodic consolidation (curator → class-level umbrellas, archive-not-delete, dry-run gate) is a reusable design for keeping a growing memory coherent instead of letting it sprawl. Tied to: **memory systems that stay useful over long horizons.**
- **The anti-learning rules (HIGH, as a *requirement*).** Any seed AI that writes to long-term memory must avoid persisting environment-specific failures and "X is broken" claims that "harden into refusals." Hermes' prompt is a ready-made checklist of what NOT to persist. Tied to: **decision-making about what to remember; avoiding self-imposed capability regressions.**
- **Progressive disclosure + manifest-cached index (MEDIUM).** Carry only metadata in the prompt; load bodies on demand; cache the index with an mtime/size manifest. Relevant to "how does a long-running agent hold a large and growing memory affordably and deterministically." Tied to: **memory retrieval, context engineering.**
- **Trajectory generation/compression as a model-improvement flywheel (MEDIUM).** `batch_runner.py` + `trajectory_compressor.py` + ShareGPT export + Atropos RL integration show the plumbing for turning agent runs into training data — the weight-level self-improvement loop our project might eventually want. Tied to: **closing the loop from behavior to model.**
- **What Hermes does NOT give us (the spine of a seed AI).** It has **no verifier / no fitness function / no "keep only if verifiably better" gate** in its core loop. Retention is an LLM judgment, edits are unvalidated, and (per #25833) the agent is "author, executor, and quality inspector" of its own knowledge. Our seed AI's defining loop is *the opposite*: propose → **test in isolation** → keep only if measurably better. So Hermes is best read as **"the memory/skill substrate you'd bolt onto an evolutionary loop,"** plus a **catalogue of the failure modes** (skill rot, drift=overfitting, conflict, advisory-only hygiene, fragile prompt-guardrails) that a *real* verifier is meant to prevent. The one place real evolutionary search appears — the `darwinian-evolver` skill — is an **external** tool (Imbue) wrapped as a playbook, and its `Organism/Evaluator(score, trainable vs. holdout failure cases, is_viable)/Mutator` schema is itself a useful reference for our verifier/candidate representation.

## 9. Signal assessment

- **Overall value: MEDIUM-HIGH** for our project. High for the memory/skill/long-horizon/orchestration dimension (the best open-source reference I've seen for an agent that *curates its own procedural knowledge*, with real, quotable prompts and a thoughtful two-tier add/curate design, plus production plumbing for running autonomously across surfaces). Lower for the *evolutionary/verification* spine of a seed AI — Hermes deliberately has no verifier and is, by its own issue tracker's admission, a self-grading loop; its main lesson there is *cautionary* (the catalogue of skill-rot/drift/conflict/advisory-hygiene failure modes that a real verifier exists to prevent).
- **Confidence: HIGH on mechanism** (read the actual code at a known SHA: the review fork, the trigger cadence, the curator prompt, the skill tool, the telemetry, the progressive-disclosure index). **HIGH on the central critique** (grounded in the project's own open issues #25833/#12877/#20595 plus the now-fixed #6051, and two engineering-serious independent analyses). **HIGH that GEPA is absent from the code** (exhaustive search; the secondary claim is unsupported).
- **What I could NOT verify:** (a) any *quantitative* effect of the learning loop on task quality — no first-party benchmarks; the "40% faster / per-run cost" figures are secondary and methodology-free. (b) The exact production trigger defaults for `memory.nudge_interval` / `skills.nudge_interval` shipped to users — I confirmed the *trigger logic* and that tests use values like 5/10, and `cli-config.yaml.example` would hold the shipped default (not fully traced). (c) The `hermes-agent-dev` skill's `references/self-improvement-loop.md` (referenced by `background_review.py`) is not present in this tarball (dev-only/unshipped), so the authors' own "PR review criteria / invariants" for the loop could not be read. (d) Whether the codeload `main` tarball is *byte-identical* to `@d1367355` — I fetched `main` and read the SHA from the API at the same time, so it is at or within hours of that commit; treat file line numbers as accurate-to-this-snapshot. (e) The GEPA/ICLR-2026 claim's origin — I can only confirm it is not in the code; I did not find a Nous source making it (it appears to originate in the DEV.to post).
